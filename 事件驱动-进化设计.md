## 1. 架构全景与核心理念

### 1.1 系统定位
“弈”不再仅仅是一个聊天界面，而是一个 **“AI 工作单元调度中心”**。用户、Agent、系统定时器三者平等地向“事件中心（Event Hub）”投递“事件（Event）”，调度器负责消费并分配角色执行。

### 1.2 三大设计原则
1. **统一接入（Unified Ingress）**：无论是用户手动点击、Agent 顿悟触发，还是凌晨定时任务，都归一化为标准的 `Event` 记录。
2. **执行可视化（Visual Execution）**：所有事件在执行时，都会生成一个 **“只读会话（Read-only Session）”**，用户可在聊天界面实时观看 AI 的每一步操作。
3. **来源可溯（Source Traceability）**：每个事件都携带 `source_type`（user/agent/system），实现全链路审计。

## 2. 数据模型定义（Schema DDL）

### 2.1 事件表（`events`）—— 核心中枢
```sql
CREATE TABLE events (
  id TEXT PRIMARY KEY,
  
  -- 来源系统
  source_type TEXT NOT NULL CHECK(source_type IN ('user', 'agent', 'system')),
  source_id TEXT,                 -- user_id / agent_id / null (system)
  source_meta TEXT,               -- JSON: { session_id, trigger, ui_action }
  
  -- 执行指派
  assigned_agent_id TEXT NOT NULL,-- 如 'sys_evolver'
  assigned_group_id TEXT,
  
  -- 定义
  type TEXT NOT NULL CHECK(type IN ('once', 'cron')),
  cron_expr TEXT,                 -- 仅 cron
  payload TEXT NOT NULL,          -- JSON: { instruction, context, ttl }
  
  -- 状态机
  status TEXT DEFAULT 'pending' CHECK(status IN ('pending', 'paused', 'running', 'success', 'failed', 'expired')),
  priority INTEGER DEFAULT 0,
  
  -- 时间审计
  scheduled_at DATETIME,
  started_at DATETIME,
  finished_at DATETIME,
  result_summary TEXT,
  error_log TEXT,
  
  -- 链式追踪
  parent_event_id TEXT,
  retry_count INTEGER DEFAULT 0,
  max_retries INTEGER DEFAULT 3,
  
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_events_status_schedule ON events(status, scheduled_at);
CREATE INDEX idx_events_source ON events(source_type, source_id);
```

### 2.2 会话表扩展（`sessions`）
```sql
ALTER TABLE sessions ADD COLUMN session_type TEXT DEFAULT 'chat'; -- 'chat' | 'event'
ALTER TABLE sessions ADD COLUMN event_id TEXT;                    -- 关联 events.id
ALTER TABLE sessions ADD COLUMN is_readonly BOOLEAN DEFAULT 0;
ALTER TABLE sessions ADD COLUMN title TEXT;                       -- 自动生成摘要标题
```

### 2.3 轨迹存储表（`trajectories`）—— 离线复盘数据源
```sql
CREATE TABLE trajectories (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  user_goal TEXT,                 -- 用户原始意图摘要
  tool_calls JSON,                -- [{ name, args, result, success, duration }]
  summary TEXT,                   -- 压缩的上下文
  success_rate REAL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 4. 核心流程实现规范（按执行顺序）

### 4.1 流程一：在线顿悟 -> 投递事件
**文件**：`apps/server/src/core/engine/sessionLoop.ts`
**触发时机**：每次内层循环结束，检测到顿悟条件时。

```typescript
// 核心逻辑伪代码
if (detected && agent.evolution.mode !== 'off') {
  const status = agent.evolution.mode === 'auto' ? 'pending' : 'paused';
  
  await eventService.create({
    source_type: 'agent',
    source_id: agent.id,
    source_meta: { session_id: session.id, trigger: detected.type },
    assigned_agent_id: 'sys_evolver',
    type: 'once',
    payload: { instruction: `分析会话 ${session.id} 的轨迹并提炼技能` },
    status: status,
    scheduled_at: new Date(),
  });

  // 通知前端
  if (status === 'paused') io.to(clientId).emit('event:require_approval');
}
```

### 4.2 流程二：离线定时复盘 -> 投递事件
**文件**：`apps/server/src/scheduler/cronRegistry.ts`
**触发时机**：每日凌晨 2:00 (Cron `0 2 * * *`)。

```typescript

    source_type: 'system',
    source_id: 'cron_daemon',
    assigned_agent_id: 'sys_evolver',

    status: 'pending',

  });
});
```

### 4.3 流程三：事件调度器消费（核心枢纽）
**文件**：`apps/server/src/event/eventScheduler.ts`
**运行机制**：每 10 秒轮询一次 `pending` 事件。

```typescript
async function poll() {
  const events = await db.query(
    `SELECT * FROM events WHERE status='pending' AND scheduled_at <= now() LIMIT 5`
  );
  for (const evt of events) {
    await executeEvent(evt);
  }
}

async function executeEvent(event: EventRecord) {
  // 1. 创建只读会话
  const sessionId = generateId();
  await db.sessions.insert({
    id: sessionId,
    agent_id: event.assigned_agent_id,
    user_id: event.creator_id,
    session_type: 'event',
    event_id: event.id,
    is_readonly: true,
    title: `📋 ${event.payload.instruction.slice(0, 30)}...`,
  });

  // 2. 通知前端（新会话出现）
  io.to(event.creator_id).emit('session:new', { sessionId, isEvent: true });

  // 3. 调用 Processor Loop（关键复用）
  await runProcessorLoop({
    sessionId,
    agentId: event.assigned_agent_id,
    systemPrompt: event.payload.instruction,
    isReadonly: true, // 强制 Bypass 策略
    initialContext: event.payload.context,
  });

  // 4. 收尾
  await eventService.update(event.id, { status: 'success', finished_at: new Date() });
  
  // 5. Cron 续期（若为定时事件）
  if (event.type === 'cron') {
    const nextRun = cronParser.parse(event.cron_expr).next().toDate();
    await eventService.create({ ...event, id: newId(), parent_event_id: event.id, scheduled_at: nextRun });
  }
}
```

## 5. 离线聚类算法（OfflineMiner）集成规范

**文件**：`apps/server/src/evolution/detectors/offlineMiner.ts`
**算法策略**：分桶（首工具+长度级） + LCS 相似度计算 + Union-Find 连通分量。

**集成到事件中的方式**：
在 `sys_evolver` 收到离线复盘事件后，它内部会调用：
```typescript
// 伪代码：sys_evolver 的执行逻辑
const rawTrajectories = await trajectoryStore.getRecent(7);
const clusters = await OfflineMiner.mine(rawTrajectories, 0.65, 3);
for (const cluster of clusters) {
  const draft = await InsightExtractor.extract(cluster);
  await SkillGenerator.generate(draft);
}
```

## 6. API 与 WebSocket 接口规范

### 6.1 后端 API（REST）

| 方法 | 路径 | 描述 | 权限 |
| :--- | :--- | :--- | :--- |
| `GET` | `/api/events` | 获取事件列表（支持筛选 source/status） | 登录用户 |
| `GET` | `/api/events/:id` | 获取事件详情（含执行日志） | 所有者 |
| `POST` | `/api/events` | **手动创建事件**（用户来源） | 登录用户 |
| `PATCH` | `/api/events/:id/status` | 更新状态（运行/暂停/重试） | 所有者 |
| `DELETE` | `/api/events/:id` | 删除事件 | 所有者 |

**手动创建事件 Body**：
```json
{
  "assigned_agent_id": "sys_curator",
  "type": "once",
  "payload": { "instruction": "整理上周的 API 文档到知识库" },
  "scheduled_at": "2026-07-05T10:00:00Z"
}
```

### 6.2 WebSocket 事件推送

| 事件名（Server -> Client） | 触发时机 | Payload |
| :--- | :--- | :--- |
| `event:require_approval` | Confirm 模式下检测到顿悟 | `{ eventId, summary }` |
| `session:new` | 调度器创建了只读会话 | `{ sessionId, title, isEvent: true }` |
| `event:status_changed` | 事件状态变化（running/success/failed） | `{ eventId, status, result_summary }` |

- **原“任务中心”** → **“事件中心”（`/events`）**。

## 9. 测试验收标准（QA Checklist）

| 测试场景 | 操作步骤 | 预期结果 | 优先级 |
| :--- | :--- | :--- | :--- |
| **Off 模式** | 设置角色为 Off，高频调用工具 | 无事件产生，无任何通知 | P0 |
| **Confirm + 投递** | 触发顿悟 | `events` 表新增记录 status=paused，前端弹出通知 | P0 |
| **Confirm + 审批** | 点击“运行” | 状态变 pending → 调度器拉起只读会话 → 聊天界面实时滚动 | P0 |
| **Auto + 自动执行** | 触发顿悟 | 事件直接 running → 只读会话自动生成，右下角 Toast 提示 | P0 |
| **手动创建事件** | 前端点击新建，选角色填指令 | API 插入记录 source_type=user，调度器按时执行 | P1 |
| **离线复盘 (Cron)** | 模拟凌晨 2 点 | 插入一条 system 来源事件，执行聚类，若产生新技能则落盘 | P1 |
| **只读隔离** | 在只读会话中尝试发消息 | 后端返回 403 错误，前端输入框禁用 | P0 |
| **聚类算法** | 插入 5 条相似轨迹（grep->awk->read） | 聚类簇大小 ≥3，提取签名正确 | P1 |
