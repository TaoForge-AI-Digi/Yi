### 杂项 性能优化
P0 — 影响稳定性和用户体验
1. Sub-agent 级联取消 当前：abort 父 session 不会自动 cancel 子 session（sub-agent）。子 session 的 bash 进程变成孤儿进程继续跑。 方案：agent/outer.ts 的 spawnAndRunSubAgent 处，将子 session 的 AbortController 注册到父 session 的 CancelScope 上。

2. MCP 服务器共享 当前：每次 sessionLoop 都重新连接 MCP 服务器，跨 session 不共享。如果多个 session 同时跑，同一台 MCP 服务器被重复连接。 方案：MCP client 池化，按 server name 复用连接，引用计数释放。

P1 — 性能
3. DB 写批处理 当前：每个 tool 结果都单独 messageStore.addMessage + sessions.update(updated_at)。多个 tool 跑完时产生大量同步 SQLite 写入。 方案：completeRun 后批量 flush 一次，或用更高效的方式合并 updated_at 更新。

4. tool.output 客户端节流 当前：每块输出都触发 Vue reactivity + DOM 更新。快速输出的命令（如 git clone 进度条）可能产生每秒几十次更新。 方案：服务端或客户端 50ms 窗口内合并输出块再渲染。

P2 — 功能完整度
5. Shell 完整输出查看 当前：截断后只保留尾部 50KB + temp 文件路径，但 UI 上没有"查看完整输出"入口。 方案：加一个 API 端点提供 temp 文件内容，ToolDetail 截断时显示"查看完整输出"链接。

6. 事件驱动调度 当前：事件轮询间隔 10s，实时性差。 方案：事件创建时主动触发调度（已有 scheduleImmediate），减少轮询延迟。

7. 长 session 上下文压缩 当前：没有 compaction。调 keepMessages 只能手动截断。 方案：context 接近 token 上限时自动压缩历史（opencode 的 compactHistory 模式）。

P3 — 运维
8. 全局并发限制 当前：无上限。大量 session 同时跑能把 LLM provider 打到 rate limit。 方案：session-runner 加全局 max concurrency，超出返回错误或排队。

9. 监控指标 当前：无。 方案：暴露 session 数量、队列长度、bash 执行时间、DB 延迟等指标。
