# Web Resilience — 网络请求容错指南

## 目标

当 `websearch` 或 `webfetch` 工具调用失败时，自动执行回退链，避免无脑重试同一操作。确保在网络不可用的情况下仍能给出有意义的结果或清晰的状态报告。

## 问题模式

在 session `mr7ti31moda2c3` 中观察到：
1. `websearch` 连续 3 次失败（fetch failed），每次都换查询词但不换策略
2. 然后尝试 `webfetch` 到新闻网站，仍未奏效
3. **耗时约 3 轮无进展，也未告知用户真实情况**

## 回退链（按优先级）

```
┌─────────────────────────────┐
│  主请求                      │
│  websearch / webfetch       │
└──────────┬──────────────────┘
           │ 失败
           ▼
┌─────────────────────────────┐
│  第 1 级：降级源             │
│  换一个数据源                │
│  (RSS / 备用搜索引擎)        │
└──────────┬──────────────────┘
           │ 失败
           ▼
┌─────────────────────────────┐
│  第 2 级：终端直连           │
│  curl/wget 目标 URL          │
│  (绕过 HTTP 工具层)          │
└──────────┬──────────────────┘
           │ 失败
           ▼
┌─────────────────────────────┐
│  第 3 级：本地缓存           │
│  读取本地缓存/快照           │
└──────────┬──────────────────┘
           │ 失败
           ▼
┌─────────────────────────────┐
│  第 4 级：告知用户           │
│  清晰说明错误+可选方案       │
└─────────────────────────────┘
```

## 具体策略

### 1. 重试策略（内置在每次请求中）

每次 web 请求应内置指数退避重试：

```
第 1 次失败 → 等待 1~1.5s
第 2 次失败 → 告知用户"换个方式试试" + 等待 2~2.5s
第 3 次失败 → 进入降级源或终端直连
```

### 2. 降级源

当首选后端失败时，尝试以下替代方案：

| 数据类型 | 首选 | 降级 1 | 降级 2 |
|---------|------|--------|--------|
| 新闻 | websearch | RSS 抓取 (bash curl) | 本地新闻缓存 |
| 通用搜索 | websearch | 切换后端 (如果配置了多个) | DDG lite |
| 网页内容 | webextract | curl + html2text | 缓存的快照 |

### 3. 终端直连（bash 方案）

当 websearch/webfetch 全部失败时，用 bash 直接 fetch：

```bash
# 新闻 RSS 示例
curl -s "https://news.google.com/rss?hl=zh-CN&gl=CN&ceid=CN:zh-Hans"

# 或直接 curl 新闻网站 + 剥离 HTML 标签
curl -s "https://news.sina.com.cn/" | python3 -c "
import sys, re, html
text = re.sub(r'<[^>]+>', '', sys.stdin.read())
print(html.unescape(text))
"
```

### 4. 本地缓存

检查本地缓存的旧数据：
- 路径：`~/.hermes/cache/web/`
- 读取最近的文件作为 fallback 内容
- 在回复中注明数据时间戳

### 5. 用户沟通模板

**第 1 次失败** → 静默重试（遵循重试策略）

**连续 2 次失败** → 告知用户并切换策略：
> "网络请求目前不太稳定，我换个方式试试..."

**连续 3 次以上失败** → 清晰报告：
> "网络似乎暂时不可用，目前无法获取最新的内容。\
> 你可以：1) 稍后再试 2) 检查网络连接 3) 或者看看本地缓存的历史记录。"

## 约束

- 不要连续发送超过 3 次相同的网络请求
- 每次重试必须有指数退避（不要在同一秒内重试）
- 第 2 次失败后必须在回复中告知用户当前状态
- 第 3 次失败后必须给用户提供可选方案
- 优先使用 bash 的 curl/wget 作为降级方案（比 webfetch 更底层）
