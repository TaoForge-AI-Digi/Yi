## 一、总览

### 1.1 项目愿景

**「弈」(Yì)** 是一个融合了 OpenCode 执行流程、Hermes 人格进化与 Flowith 知识花园 **理念** 的 AI Agent 系统。其核心特征：

- **有灵魂**：每个 Agent 拥有独立的身份（Soul）、记忆与人格，内置三个经典角色。
- **有策略**：用户可手动切换 Plan/Ask/Bypass 执行模式。
- **有分身**：主 Agent 可拉起同组的多个 Sub-agent 并行处理子任务。
- **有记忆**：知识以"典籍"形态存储，通过语义检索与图谱扩散召回。**（[Phase: 知识库]）**
- **会进化**：成功的执行轨迹可沉淀为 Skill 或新典籍。进化行为由**事件驱动**，用户可管可控。
- **有事件**：**[+]** 所有系统行为（进化、知识整理、定时任务）统一建模为"事件（Event）"，在**事件中心**集中管理，在执行时生成**只读会话**供用户实时观测。
- **有温度**：**[+]** 用户与小师妹对话，她会在需要时拉起易大师或结先生，三角协作让能力增长、知识沉淀与日常陪伴融为一体。

### 1.2 核心概念定义

| 概念 | 定义 | 变更 |
|:---|:---|:---|
| **Agent** | 拥有独立身份（Soul）、记忆、工具集、技能库和知识绑定的执行单元。 | — |
| **Main Agent** | 用户可直接对话的入口 Agent（`role = main` 或 `both`）。 | — |
| **Sub-agent** | 被其他 Agent 拉起的子执行单元（`role = sub` 或 `both`），拥有独立上下文。 | — |
| **Group** | Agent 的逻辑归属域。同组 Agent 可相互委托，跨组不可见。 | — |
| **Strategy** | 用户手动选择的执行策略：`Plan`（只读）、`Ask`（需确认）、`Bypass`（直接执行）。 | — |
| **Soul** | Agent 的人格定义文件（`SOUL.md`），固定存在于系统提示词顶层。 | — |
| **Skill** | 渐进式披露的标准化操作手册（`SKILL.md`），按需加载。 | — |
| **[+] Event** | **事件**——系统中的一个工作单元。由用户、Agent 或系统定时器发起，由指定 Agent 执行，整个过程可追溯、可观测。 | 新增 |
| **[+] Event Source** | **事件来源**——触发事件的主体，取值为 `user`（用户手动）、`agent`（Agent 自主触发）、`system`（系统定时/启动）。 | 新增 |

| **Sutra (典籍)** | **[Phase: 知识库]** 知识的最小原子单元，包含摘要、向量、原文位置与经络链接。 | — |
| **Meridian Link (经络链接)** | **[Phase: 知识库]** 典籍之间的语义关联（因果、参考、演进等）。 | — |

## 二、身份引擎

### 2.1 Agent 注册模型 **[M]**

```pseudocode
AgentRegistration:
    id: string                    // 唯一标识
    role: "main" | "sub" | "both" // Agent 角色能力
    group: string                 // 所属组名
    hidden: boolean DEFAULT false // [+ 新增] 是否对普通用户隐藏
    
    // 身份绑定
    identity:
        soul_path: string         // SOUL.md 文件路径
        memory_path: string       // MEMORY.md 文件路径
        user_memory_path: string  // USER.md 文件路径 (可选)
        
    // 能力绑定（权限 L1）
    bindings:
        skills: list<string>      // Skill 文件路径列表
        tools: list<ToolBinding>  // 工具绑定列表（含约束）
        knowledge: KnowledgeBinding? // [可选] 藏经阁连接配置
        
    // 策略默认值
    default_strategy: "Plan" | "Ask" | "Bypass"
    
    // [+ 新增] 进化控制
    evolution:
        mode: "off" | "confirm" | "auto"   // 进化三态开关
        auto_quota_daily: number DEFAULT 3  // 自动模式下每日生成上限
```

    agent = AGENT_REGISTRY.get(agent_id)

## 三、策略引擎

### 3.1 三种策略模式

| 模式 | 用户指令 | 允许操作 | 禁止操作 |
|:---|:---|:---|:---|
| **Plan** | `/plan` | 只读工具、Skill 索引、`sutra_query` [Phase:知识库] | 所有写入/执行类工具、`delegate_task`、委托工具 |
| **Ask** | `/ask` | 所有工具可调用，写入/执行类需用户确认 | 无 |
| **Bypass** | `/bypass` | 所有工具无条件放行 | 无（除非强制确认） |

> **注意**：对于**只读事件会话**，即使策略为 `Ask`，也会被强制覆盖为 `Bypass` 以避免阻塞。（详见 §7.4）

### 3.2 用户命令

```pseudocode
// 用户在输入框输入特殊命令切换策略
// 当前策略显示在聊天界面顶部

COMMANDS:
    /plan     → 切换为 Plan 模式（只读）
    /ask      → 切换为 Ask 模式（需确认）
    /bypass   → 切换为 Bypass 模式（直接执行）
    /status   → 显示当前策略与 Agent 信息
```

### 3.3 策略状态管理

```pseudocode
CLASS SessionState:
    current_strategy: "Plan" | "Ask" | "Bypass"
    user_set_at: timestamp
    
    // [+ 新增] 只读会话强制覆盖
    is_readonly_override: boolean DEFAULT false
    
    FUNCTION can_execute_tool(tool_call):
        IF is_readonly_override:
            RETURN PASS  // 跳过所有策略拦截
        
        SWITCH current_strategy:
            CASE "Plan":
                IF tool_call.is_readonly OR tool_call.is_skill_load:
                    RETURN PASS
                ELSE:
                    RETURN BLOCKED("Plan 模式下禁止执行写入/执行操作")
            CASE "Ask":
                IF tool_call.is_dangerous:
                    RETURN REQUIRE_CONFIRM
                ELSE:
                    RETURN PASS
            CASE "Bypass":
                IF tool_call.has_require_confirm_even_in_bypass:
                    RETURN REQUIRE_CONFIRM
                ELSE:
                    RETURN PASS
```

### 4.3 内层循环 **[M]**

```pseudocode
FUNCTION inner_loop(session, prompt, is_readonly):
    IF is_readonly:
        session.state.is_readonly_override = true
    
    session.history.append({ role: "user", content: prompt })
    
    WHILE NOT session.should_exit:
        response = llm.invoke(session.system_prompt + session.history)
        
        IF response.has_tool_call:
            tool_call = response.tool_call
            
            // 处理委托工具
            IF tool_call.name IN ["delegate_task", "ask_master_yi", "ask_mr_jie"]:
                sub_signal = handle_sub_agent_request(tool_call)
                session.sub_agent_results.append(sub_signal)
                CONTINUE
            
            // 两阶段校验
            validation = validate_tool_execution(tool_call, session)
            IF validation == BLOCKED:
                session.history.append({ role: "tool", error: validation.message })
                CONTINUE
            ELSE IF validation == REQUIRE_CONFIRM AND NOT is_readonly:
                // 向用户发起确认请求，等待响应
                confirm = await request_user_confirm(tool_call)
                IF NOT confirm:
                    session.history.append({ role: "tool", error: "用户已取消" })
                    CONTINUE
            
            // 执行工具
            result = execute_tool(tool_call)
            session.history.append({ role: "tool", content: result })
        
        ELSE:
            // LLM 直接返回文本，检测是否为最终答案
            IF response.is_final:
                session.final_answer = response.content
                session.should_exit = true
            ELSE:
                session.history.append({ role: "assistant", content: response.content })
        
        // Doom Loop 检测
        IF detect_doom_loop(session):
            session.should_exit = true
            session.final_answer = "检测到死循环，已终止"
    
    RETURN session.final_answer
```

## 七、事件引擎 **[+] — 全新模块**

### 7.1 设计理念

三大原则：
1. **统一接入**：用户、Agent、系统三者平等地向事件中心投递事件。
2. **来源可溯**：每个事件携带 `source_type`（user/agent/system），实现全链路审计。
3. **执行可视**：每个事件在执行时生成**只读会话**，用户可在聊天界面实时观测每一步操作。

### 7.2 事件数据模型

```pseudocode
Event:
    id: string
    
    // --- 事件来源 (新增核心字段) ---
    source_type: "user" | "agent" | "system"
    source_id: string?            // user_id / agent_id / null (system)
    source_meta: JSON             // { session_id, trigger_type, ui_action }
    
    // --- 执行指派 ---
    assigned_agent_id: string     // 实际干活的角色（如 master_yi, mr_jie）
    assigned_group_id: string?    // 可指定群组
    
    // --- 事件定义 ---
    type: "once" | "cron"
    cron_expr: string?            // 仅 type="cron" 时有效
    payload: JSON                 // { instruction, context, ttl }
    
    // --- 状态机 ---
    status: "pending" | "paused" | "running" | "success" | "failed" | "expired"
    priority: int DEFAULT 0
    
    // --- 时间审计 ---
    scheduled_at: datetime
    started_at: datetime?
    finished_at: datetime?
    result_summary: string?
    error_log: string?
    
    // --- 链式追踪 ---
    parent_event_id: string?      // 用于事件链（复盘事件产生的新进化事件）
    retry_count: int DEFAULT 0
    max_retries: int DEFAULT 3
    
    created_at: datetime
```

### 7.3 事件来源规范

| `source_type` | `source_id` 含义 | `source_meta` 内容 | 典型场景 |
|:---|:---|:---|:---|
| **`user`** | 用户 ID | `{ "ui_action": "manual_create", "cron_preview": "..." }` | 用户在事件中心手动创建；在 Confirm 模式下点击"批准" |
| **`agent`** | Agent ID | `{ "session_id": "S-123", "trigger": "high_freq_tools" }` | 会话中检测到顿悟；Sub-agent 执行完任务后请求归档 |
| **`system`** | `'cron_daemon'` / `'bootstrap'` | `{ "schedule": "daily_2am", "lookback_days": 7 }` | 凌晨定时离线复盘；系统启动时初始化整理 |

### 7.4 事件调度器

```pseudocode
// 轮询调度器：每 10 秒扫描一次
FUNCTION event_scheduler_poll():
    events = db.query(
        "SELECT * FROM events WHERE status='pending' AND scheduled_at <= now()"
        + " ORDER BY priority DESC, created_at ASC LIMIT 5"
    )
    
    FOR EACH event IN events:
        execute_event(event)

FUNCTION execute_event(event):
    // 1. 创建只读会话（与普通聊天同源）
    session_id = generate_id()
    session = create_session(
        id = session_id,
        agent_id = event.assigned_agent_id,
        user_id = event.creator_id,
        session_type = "event",      // 标记为事件会话
        event_id = event.id,
        is_readonly = true,          // 关键标志：只读模式
        title = "📋 事件: " + event.payload.instruction[:30] + "..."
    )
    
    // 2. 通知前端（新会话出现在侧边栏）
    io.to(event.creator_id).emit("session:new", {
        sessionId: session_id,
        title: session.title,
        isEvent: true
    })
    
    // 3. 插入系统开场消息
    session.history.append({
        role: "system",
        content: "🔄 **事件执行中**\n来源: " + event.source_type
                + "\n指令: " + event.payload.instruction
    })
    
    // 4. 调用处理器循环（只读模式）
    result = inner_loop(
        session = session,
        prompt = event.payload.instruction,
        is_readonly = true          // 强制 Bypass，禁止用户确认弹窗
    )
    
    // 5. 更新事件状态
    db.events.update(event.id, {
        status: "success",
        finished_at: now(),
        result_summary: result.summary
    })
    
    // 6. 处理 Cron 续期
    IF event.type == "cron":
        next_run = cron_next(event.cron_expr)
        create_event({
            ...event,
            id: new_id(),
            parent_event_id: event.id,
            status: "pending",
            scheduled_at: next_run
        })
```

### 7.5 只读会话（Read-only Session）

**核心机制**：
- 事件会话与普通聊天会话共享同一套消息渲染引擎（流式输出、Tool Call 卡片、思维链展示）。

## 八、权限与进化控制 **[M]**

### 8.1 两阶段校验

| 阶段 | 归属引擎 | 检查内容 | 数据来源 |
|:---|:---|:---|:---|
| **L1: 身份绑定** | 身份引擎 | 工具是否在 `tool_bindings` 中 + 约束条件是否满足 | `AgentRegistration.bindings.tools` |
| **L2: 策略拦截** | 策略引擎 | 当前策略是否允许调用该类工具 | `SessionState.current_strategy` + `DANGEROUS_TOOLS` |

> **[+ 新增]** 对于 **只读事件会话**，L2 策略拦截被强制覆盖为 `Bypass`，不弹出用户确认对话框。

### 8.2 两阶段校验实现

```pseudocode
FUNCTION validate_tool_execution(tool_call, session):
    // ========== L1: 身份引擎 - 工具绑定校验 ==========
    binding = session.identity.tools.find(t => t.name == tool_call.name)
    IF binding == null:
        RETURN BLOCKED("工具 '{tool_call.name}' 未注册")
    
    // 约束校验
    IF not check_allowed_paths(binding, tool_call):
        RETURN BLOCKED("路径不在允许范围内")
    IF not check_allowed_commands(binding, tool_call):
        RETURN BLOCKED("命令不在允许范围内")
    // ... 其他约束校验
    
    // ========== L2: 策略引擎 - 模式拦截 ==========
    // [+ 新增] 只读事件会话 → 强制 Bypass
    IF session.state.is_readonly_override:
        RETURN PASS
    
    strategy = session.state.current_strategy
    SWITCH strategy:
        CASE "Plan":
            IF tool_call.name IN DANGEROUS_TOOLS:
                RETURN BLOCKED("Plan 模式下禁止执行 '{tool_call.name}'")
        CASE "Ask":
            IF tool_call.name IN DANGEROUS_TOOLS:
                RETURN REQUIRE_CONFIRM
        CASE "Bypass":
            IF binding.constraints.require_confirm_even_in_bypass:
                RETURN REQUIRE_CONFIRM
    
    RETURN PASS
```

| `confirm` | 检测到顿悟 → 创建 `paused` 事件 → 事件中心待审批 | 收到通知，手动批准后执行 |
| `auto` | 检测到顿悟 → 创建 `pending` 事件 → 调度器自动执行 | 右下角 Toast 轻提示 |
