## 一、总览

### 1.1 项目愿景

**「弈」(Yì)** 是一个融合了 OpenCode 执行流程、Hermes 人格进化与 Flowith 知识花园 **理念** 的 AI Agent 系统。其核心特征：

- **有灵魂**：每个 Agent 拥有独立的身份（Soul）、记忆与人格。
- **有策略**：用户可手动切换 Plan/Ask/Bypass 执行模式。
- **有分身**：主 Agent 可拉起同组的多个 Sub-agent 并行处理子任务。
- **有记忆**：知识以"典籍"形态存储，通过语义检索与图谱扩散召回。**（[Phase: 知识库]）**
- **会进化**：成功的执行轨迹可沉淀为 Skill 或新典籍。进化行为由**事件驱动**，用户可管可控。
- **有事件**：**[+]** 所有系统行为（进化、知识整理、定时任务）统一建模为“事件（Event）”，在**事件中心**集中管理，在执行时生成**只读会话**供用户实时观测。

### 1.2 核心概念定义

| 概念 | 定义 | 变更 |
|:---|:---|:---|
| **Agent** | 拥有独立身份（Soul）、记忆、工具集、技能库和知识绑定的执行单元。 | — |
| **Main Agent** | 用户可直接对话的入口 Agent（`role = main` 或 `both`）。 | — |
| **Sub-agent** | 被其他 Agent 拉起的子执行单元（`role = sub` 或 `both`），拥有独立上下文。 | — |
| **Group** | Agent 的逻辑归属域。同组 Agent 可相互委托，跨组不可见。 | — |
| **Strategy** | 用户手动选择的执行策略：`Plan`（只读）、`Ask`（需确认）、`Bypass`（直接执行）。 | — |
| **Soul** | Agent 的人格定义文件（`SOUL.md`），固定存在于系统提示词顶层。 | — |
| **Skill** | 渐进式披露的标准化操作手册（`SKILL.md`），按需加载。 | — |
| **[+] Event** | **事件**——系统中的一个工作单元。由用户、Agent 或系统定时器发起，由指定 Agent 执行，整个过程可追溯、可观测。 | 新增 |
| **[+] Event Source** | **事件来源**——触发事件的主体，取值为 `user`（用户手动）、`agent`（Agent 自主触发）、`system`（系统定时/启动）。 | 新增 |
| **[+] Evolution Mode** | **进化模式**——每个 Agent 独立的进化行为开关：`off`（关闭）、`confirm`（需确认）、`auto`（自动执行）。 | 新增 |
| **Sutra (典籍)** | **[Phase: 知识库]** 知识的最小原子单元，包含摘要、向量、原文位置与经络链接。 | — |
| **Meridian Link (经络链接)** | **[Phase: 知识库]** 典籍之间的语义关联（因果、参考、演进等）。 | — |

## 三、策略引擎

### 3.1 三种策略模式（与原文一致）

| 模式 | 用户指令 | 允许操作 | 禁止操作 |
|:---|:---|:---|:---|
| **Plan** | `/plan` | `read`, `grep`, `glob`, `lsp`, `skill_load`(索引), `sutra_query` [Phase:知识库] | 所有写入 (`write/edit/delete`)、执行 (`bash`/`execute`)、`delegate_task` |
| **Ask** | `/ask` | 所有工具**均可调用**，但在调用写入/执行类工具前，**必须向用户请求确认** | 无（但确认环节可能被用户拒绝） |
| **Bypass** | `/bypass` | 所有工具无条件放行，无需确认 | 无（除非工具绑定了 `require_confirm_even_in_bypass`） |

> **注意**：`Bypass` 策略仅适用于**普通会话中的工具执行**。对于 **只读事件会话**，即使策略为 `Ask`，也会被引擎强制覆盖为 `Bypass` 以避免阻塞。（详见 § 7.4）

### 4.2 外层循环 **[M]**

```pseudocode
FUNCTION session_loop(agent_id, user_input):
    // 1. 验证 Agent 可作主
    agent = validate_agent_for_user(agent_id)
    
    // 2. 加载身份
    identity = load_identity(agent_id)
    
    // 3. 初始化会话状态
    session = create_session(
        agent = agent,
        identity = identity,
        strategy = identity.default_strategy,
        history = []
    )
    session.evolution_mode = agent.evolution.mode  // [+ 新增]
    
    // 4. [可选] 初始化藏经阁
    IF identity.knowledge_config IS NOT NULL:
        keeper = get_sutra_keeper(identity.knowledge_config)
    ELSE:
        keeper = NoOpSutraKeeper()
    
    // 5. 主循环
    WHILE session.is_active:
        // ... 原有逻辑 ...
        
        // [+ 重构] 后处理：顿悟检测 → 投递事件
        IF session.completed_successfully:
            trigger_epiphany(
                trajectory = session.trajectory,
                agent_id = agent_id,
                evolution_mode = session.evolution_mode,
                keeper = keeper
            )
        // trigger_epiphany 不再直接生成 Skill，而是创建 Event
    
    RETURN session.final_answer
```

## 六、进化引擎 **[M] — 重大重构**

### 6.1 进化三态开关（Evolution Mode）

每个 Agent 拥有独立的进化控制开关，由用户在角色设置中配置：

| 模式 | 检测机制 | 事件创建 | 用户交互 | 适用场景 |
|:---|:---|:---|:---|:---|
| **`off`** | ❌ **完全跳过**（不计数、不检测、不消耗 Token） | 不创建 | 无感知 | 稳定运行的生产 Agent，不希望产生额外开销 |
| **`confirm`** | ✅ **正常检测** | 创建 Event，`status = 'paused'` | 用户在事件中心点击“运行”后才执行 | 需要用户监督质量的场景（推荐默认） |
| **`auto`** | ✅ **正常检测** | 创建 Event，`status = 'pending'` | 调度器自动执行，完成后 Toast 通知 | 高信任度场景，用户不希望被打断 |

### 6.2 顿悟触发条件（与原文一致）

```pseudocode
FUNCTION should_trigger_epiphany(trajectory):
    IF trajectory.tool_call_count >= 3 AND trajectory.success:
        RETURN True
    IF trajectory.has_user_correction:
        RETURN True
    IF trajectory.has_self_correction:
        RETURN True
    RETURN False
```

### 6.3 顿悟执行流程 **[M — 重构为事件投递]**

```pseudocode
FUNCTION trigger_epiphany(trajectory, agent_id, evolution_mode, keeper):
    // 1. 检查进化模式
    IF evolution_mode == "off":
        RETURN  // 完全静默，不执行任何操作
    
    // 2. 触发条件检测
    IF NOT should_trigger_epiphany(trajectory):
        RETURN
    
    // 3. 确定事件状态
    event_status = (evolution_mode == "auto") ? "pending" : "paused"
    
    // 4. 创建进化事件
    event_id = event_engine.create_event({
        source_type: "agent",
        source_id: agent_id,
        source_meta: {
            session_id: trajectory.session_id,
            trigger_type: trajectory.type,
            tool_sequence: trajectory.tool_sequence
        },
        assigned_agent_id: "sys_evolver",  // 固定的系统角色
        type: "once",
        payload: {
            instruction: "分析会话执行轨迹，提炼可复用技能",
            session_id: trajectory.session_id,
            trajectory_summary: trajectory.summary
        },
        status: event_status,
        scheduled_at: now()
    })
    
    // 5. 通知前端
    IF evolution_mode == "confirm":
        io.emit("event:require_approval", {
            event_id: event_id,
            summary: "发现可沉淀经验，请前往事件中心审批"
        })
    ELSE:  // auto
        io.emit("event:auto_created", {
            event_id: event_id,
            summary: "已自动创建进化事件"
        })
    
    RETURN event_id
```

## 七、事件引擎 **[+] — 全新模块**

### 7.1 设计理念

事件引擎是“弈”v4.0 的核心基础设施，将**所有系统行为**（进化、知识整理、用户手动任务、定时维护）统一建模为 **“事件（Event）”**。

**三大原则**：
1. **统一接入**：用户、Agent、系统三者平等地向事件中心投递事件。
2. **来源可溯**：每个事件携带 `source_type`（user/agent/system），实现全链路审计。
3. **执行可视**：每个事件在执行时生成**只读会话**，用户可在聊天界面实时观测每一步操作。

### 7.2 事件数据模型

```pseudocode
Event:
    id: string
    
    // --- 事件来源 (新增核心字段) ---
    source_type: "user" | "agent" | "system"
    source_id: string?            // user_id / agent_id / null (system)
    source_meta: JSON             // { session_id, trigger_type, ui_action }
    
    // --- 执行指派 ---
    assigned_agent_id: string     // 实际干活的角色（如 sys_evolver）
    assigned_group_id: string?    // 可指定群组
    
    // --- 事件定义 ---
    type: "once" | "cron"
    cron_expr: string?            // 仅 type="cron" 时有效
    payload: JSON                 // { instruction, context, ttl }
    
    // --- 状态机 ---
    status: "pending" | "paused" | "running" | "success" | "failed" | "expired"
    priority: int DEFAULT 0
    
    // --- 时间审计 ---
    scheduled_at: datetime
    started_at: datetime?
    finished_at: datetime?
    result_summary: string?
    error_log: string?
    
    // --- 链式追踪 ---
    parent_event_id: string?      // 用于事件链（复盘事件产生的新进化事件）
    retry_count: int DEFAULT 0
    max_retries: int DEFAULT 3
    
    created_at: datetime
```

### 7.3 事件来源规范

| `source_type` | `source_id` 含义 | `source_meta` 内容 | 典型场景 |
|:---|:---|:---|:---|
| **`user`** | 用户 ID | `{ "ui_action": "manual_create", "cron_preview": "..." }` | 用户在事件中心手动创建；在 Confirm 模式下点击“批准” |
| **`agent`** | Agent ID | `{ "session_id": "S-123", "trigger": "high_freq_tools" }` | 会话中检测到顿悟；Sub-agent 执行完任务后请求归档 |
| **`system`** | `'cron_daemon'` / `'bootstrap'` | `{ "schedule": "daily_2am", "lookback_days": 7 }` | 凌晨定时离线复盘；系统启动时初始化整理 |

### 7.4 事件调度器

```pseudocode
// 轮询调度器：每 10 秒扫描一次
FUNCTION event_scheduler_poll():
    events = db.query(
        "SELECT * FROM events WHERE status='pending' AND scheduled_at <= now()"
        + " ORDER BY priority DESC, created_at ASC LIMIT 5"
    )
    
    FOR EACH event IN events:
        execute_event(event)

FUNCTION execute_event(event):
    // 1. 创建只读会话（与普通聊天同源）
    session_id = generate_id()
    session = create_session(
        id = session_id,
        agent_id = event.assigned_agent_id,
        user_id = event.creator_id,
        session_type = "event",      // 标记为事件会话
        event_id = event.id,
        is_readonly = true,          // 关键标志：只读模式
        title = "📋 事件: " + event.payload.instruction[:30] + "..."
    )
    
    // 2. 通知前端（新会话出现在侧边栏）
    io.to(event.creator_id).emit("session:new", {
        sessionId: session_id,
        title: session.title,
        isEvent: true
    })
    
    // 3. 插入系统开场消息
    session.history.append({
        role: "system",
        content: "🔄 **事件执行中**\n来源: " + event.source_type
                + "\n指令: " + event.payload.instruction
    })
    
    // 4. 调用处理器循环（只读模式）
    result = inner_loop(
        session = session,
        prompt = event.payload.instruction,
        is_readonly = true          // 强制 Bypass，禁止用户确认弹窗
    )
    
    // 5. 更新事件状态
    db.events.update(event.id, {
        status: "success",
        finished_at: now(),
        result_summary: result.summary
    })
    
    // 6. 处理 Cron 续期
    IF event.type == "cron":
        next_run = cron_next(event.cron_expr)
        create_event({
            ...event,
            id: new_id(),
            parent_event_id: event.id,
            status: "pending",
            scheduled_at: next_run
        })
```

### 7.5 只读会话（Read-only Session）

**核心机制**：
- 事件会话与普通聊天会话共享同一套消息渲染引擎（流式输出、Tool Call 卡片、思维链展示）。
- **区别**：输入框禁用，顶部显示 **“🔒 只读模式”** 横幅。
- 后端 API 对 `is_readonly = true` 的会话**拒绝所有用户消息**（返回 403）。

```pseudocode
// 内层循环只读适配
FUNCTION inner_loop(session, prompt, is_readonly):
    IF is_readonly:
        // 强制覆盖策略为 Bypass
        session.strategy_override = "Bypass"
        // 禁止所有用户确认弹窗
        session.disable_confirm_dialog = True
    
    // 其余逻辑（LLM 推理、工具调用、流式输出）完全不变
    // 所有输出通过 Socket.IO 广播到该 session_id 房间
```

## 八、权限与进化控制 **[M]**

### 8.1 权限模型：两阶段校验

**「弈」不设独立的 `permission` 层。** 权限控制合并为**两阶段硬校验**：

| 阶段 | 归属引擎 | 检查内容 | 数据来源 |
|:---|:---|:---|:---|
| **L1: 身份绑定** | 身份引擎 | 工具是否在 `tool_bindings` 中 + 约束条件是否满足 | `AgentRegistration.bindings.tools` |
| **L2: 策略拦截** | 策略引擎 | 当前策略是否允许调用该类工具 | `SessionState.current_strategy` + `DANGEROUS_TOOLS` |

> **[+ 新增]** 对于 **只读事件会话**，L2 策略拦截被强制覆盖为 `Bypass`，不弹出用户确认对话框。

### 8.2 两阶段校验实现

```pseudocode
FUNCTION validate_tool_execution(tool_call, session):
    // ========== L1: 身份引擎 - 工具绑定校验 ==========
    // ... 与原文一致，此处省略 ...
    
    // ========== L2: 策略引擎 - 模式拦截 ==========
    // [+ 新增] 只读事件会话 → 强制 Bypass
    IF session.is_readonly:
        // 跳过所有用户确认，直接放行
        RETURN pass
    
    strategy = session.state.current_strategy
    
    // ... 原有 Plan/Ask/Bypass 逻辑 ...
```

### 8.3 进化三态开关（与 §6.1 定义一致）

每个 Agent 的进化控制由 `AgentRegistration.evolution.mode` 管理：

| 模式 | 行为 | 用户感知 |
|:---|:---|:---|
| `off` | 顿悟检测完全跳过，不消耗任何资源 | 无感知 |
| `confirm` | 检测到顿悟 → 创建 `paused` 事件 → 事件中心待审批 | 收到通知，手动批准后执行 |
| `auto` | 检测到顿悟 → 创建 `pending` 事件 → 调度器自动执行 | 右下角 Toast 轻提示 |

### C.3 关键 SQL DDL

```sql
-- 事件表
CREATE TABLE events (
  id TEXT PRIMARY KEY,
  source_type TEXT NOT NULL CHECK(source_type IN ('user', 'agent', 'system')),
  source_id TEXT,
  source_meta TEXT,
  assigned_agent_id TEXT NOT NULL,
  assigned_group_id TEXT,
  type TEXT NOT NULL CHECK(type IN ('once', 'cron')),
  cron_expr TEXT,
  payload TEXT NOT NULL,
  status TEXT DEFAULT 'pending' CHECK(status IN ('pending','paused','running','success','failed','expired')),
  priority INTEGER DEFAULT 0,
  scheduled_at DATETIME,
  started_at DATETIME,
  finished_at DATETIME,
  result_summary TEXT,
  error_log TEXT,
  parent_event_id TEXT,
  retry_count INTEGER DEFAULT 0,
  max_retries INTEGER DEFAULT 3,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_events_status_schedule ON events(status, scheduled_at);
CREATE INDEX idx_events_source ON events(source_type, source_id);

-- 会话表扩展
ALTER TABLE sessions ADD COLUMN session_type TEXT DEFAULT 'chat';
ALTER TABLE sessions ADD COLUMN event_id TEXT;
ALTER TABLE sessions ADD COLUMN is_readonly BOOLEAN DEFAULT 0;
ALTER TABLE sessions ADD COLUMN title TEXT;

-- 轨迹表
CREATE TABLE trajectories (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  agent_id TEXT NOT NULL,
  user_goal TEXT,
  tool_calls JSON,
  summary TEXT,
  success_rate REAL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```
