# Yi-Lin 工具优化计划

基于 opencode 参考实现的分析，按优先级排列。

---

## P0 — 立即优化

### 1. truncate 服务（新建）
**问题**: inner.ts 的 `truncateToolOutput` 截断后数据不可回溯。LLM 看不到完整输出。

**方案**: 新建 `src/tools/truncate/` 服务层，统一管理工具输出截断：
- 默认 50KB / 2000 行，改 inner.ts 引用此服务
- 溢出时写入 `data/tool-output/<tool>_<timestamp>.log`
- 返回内容末尾附 `(完整输出已保存至 data/tool-output/xxx.log，可用 read/grep 回溯)`
- 定期清理（7 天过期）

**参考**: opencode `packages/opencode/src/tool/truncate.ts`

---

### 2. grep 换 ripgrep
**问题**: 纯 JS 递归不忽略 `node_modules`，大项目（如 Yi-Lin 自身）遍历数万文件，直接卡死。

**方案**: 使用 ripgrep：
```typescript
import { execSync } from 'child_process'
function grepRipgrep(pattern: string, dir: string, include?: string): string {
  const includeFlag = include ? `-g "${include}"` : ''
  const result = execSync(
    `rg -n --no-heading ${includeFlag} "${pattern}" "${dir}"`,
    { encoding: 'utf-8', maxBuffer: 10 * 1024 * 1024, timeout: 30000 }
  )
  return result || 'No matches found'
}
```
- 自动忽略 `.git`、`node_modules`、`target`、`dist`、`build`
- 结果上限 200 条，超出截断并提示
- 可选 `include` 参数（文件类型过滤，如 `*.ts`）

**参考**: opencode `packages/opencode/src/tool/grep.ts`

### 3. 工具参数 Zod Schema 校验（新建服务）
**问题**: 当前每个工具手写 `args.xxx || ''` 做参数提取，无类型校验。无效参数（如 URL 格式错误、超时值超出范围）在运行时才暴露，且错误信息不友好。

**方案**: 引入 `zod` 做轻量级运行时校验层：

**Step 1: 安装**
```bash
npm install zod
```

**Step 2: 新建 `src/tools/validate.ts`**
```typescript
import { z } from 'zod'

export function validate<T>(schema: z.ZodType<T>, args: Record<string, string>, toolName: string): T {
  // args 来自 LLM，全是 string，需转型
  const parsed = schema.safeParse(args)
  if (!parsed.success) {
    const issues = parsed.error.issues.map(i => `  ${i.path.join('.')}: ${i.message}`).join('\n')
    throw new Error(`[${toolName}] 参数校验失败:\n${issues}`)
  }
  return parsed.data
}
```

**Step 3: 逐步迁移各工具**（从 webfetch 开始）：
```typescript
// 改前
const url = args.url || ''
const format = (args.format || 'markdown') as ...

// 改后
const Input = z.object({
  url: z.string().url('必须是合法 URL'),
  format: z.enum(['text', 'markdown', 'html']).default('markdown'),
  readable: z.enum(['true', 'false']).default('false'),
  timeout: z.string().optional(),
})
const input = validate(Input, args, 'webfetch')
```

**迁移计划**:
| 工具 | 优先级 | 说明 |
|---|---|---|
| webfetch | P0 | 已有 format/readable/timeout 参数，最需要校验 |
| bash | P0 | command 不能为空 |
| read | P0 | path 不能为空，offset/limit 需为数字 |
| edit | P1 | path/oldString/newString 验证 |
| write | P1 | path/content 验证 |
| websearch | P1 | query 验证 |
| grep | P2 | pattern 验证 |
| glob | P2 | pattern 验证 |
| delegate_task | P2 | task/target_character_id 验证 |

**参考**: opencode `packages/core/src/tool/webfetch.ts` 中的 `Schema.Struct` 用法

## P1 — 重要优化

### 4. read 分页 + 模糊文件名提示
**问题**: 当前一次性读整个文件，>1MB 直接拒掉。不支持 offset/limit 分页。

**方案**:
- 新增 `offset`、`limit` 参数
- 文件 >50KB 时自动启用分页模式
- 返回格式：每行加行号前缀 `<n>: <content>`
- 文件不存在时，扫描同目录文件名做模糊匹配，给出 "Did you mean?" 建议（最多 3 个）

```typescript
parameters: {
  path: { type: 'string' },
  offset: { type: 'number', description: '起始行号（1-based，可选）' },
  limit: { type: 'number', description: '最大行数（可选，默认 2000）' },
}
```

**参考**: opencode `packages/opencode/src/tool/read.ts`

---

### 5. edit 模糊匹配级联
**问题**: LLM 生成的 `oldString` 常因缩进/空格/换行差异导致精确匹配失败。当前只有 `indexOf`。

**方案**: 在精确匹配失败后，按顺序尝试以下 fallback：
1. **LineTrimmed** — 逐行 trim 后对比
2. **WhitespaceNormalized** — 空白字符归一化后对比
3. **IndentationFlexible** — 忽略缩进差异
4. **ContextAware** — 取首尾行做锚点，中间只要 50% 行匹配即可

**参考**: opencode `packages/opencode/src/tool/edit.ts`（9 级级联）

---

### 6. websearch 增加 fallback 链
**问题**: DuckDuckGo Lite HTML 刮削不稳定。一次搜索只尝试一个数据源。

**方案**: 依次尝试多个搜索后端：
1. `SEARCH_API_URL` 环境变量
2. DuckDuckGo HTML 版
3. Google 抓取（最后 fallback）

每个后端超时 10s，总计 30s。结果上限 8 条。

## P2 — 渐进优化

### 7. write BOM + 冲突检测
**问题**: 直接 `writeFileSync` 覆盖，多 session 同时写同一文件可能冲突。不处理 BOM。

**方案**:
- 写入前读原文件，检测 UTF-8 BOM 保留
- 实现 `writeIfUnchanged`：先读文件 mtime + 内容 hash，写入时检查未被其他 session 修改

### 8. bash 输出溢出时存可回溯文件
**问题**: 已存 `.bash-output/` 但 inner.ts 截断后路径丢失。

**方案**: 集成到 truncate 服务，统一管理。返回值嵌入文件路径供回溯。

### 9. glob 结果上限
**问题**: 大目录下 glob 可能返回数千条结果。

**方案**: 新增 `limit` 参数（默认 100），超出截断并提示。
