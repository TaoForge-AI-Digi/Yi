## 一、藏经阁概述

### 1.1 定位

藏经阁是「弈」系统的**长期记忆与知识沉淀层**。它使 Agent 能够：

- **取经**：在每次推理前，根据用户问题检索相关典籍摘要，注入 System Prompt。
- **引用**：在需要详细上下文时按需加载典籍全文，并**追溯到原始文档的精确位置**。
- **编纂**：将外部文档、对话轨迹或顿悟产物转化为结构化的典籍，融入知识库。
- **贯通**：通过“经络链接”自动建立典籍之间的因果、参考、演进等语义关系，形成知识网络。

藏经阁是**可选的**：当 Agent 未绑定 `knowledge` 配置时，核心引擎使用 `NoOpSutraKeeper`，所有知识操作静默返回空，不影响主流程。

### 1.2 与核心引擎的交互

核心引擎通过 `ISutraKeeper` 接口调用，不感知底层实现。交互发生在两个位置：

1. **外层循环开始前**：`keeper.query(user_input, agent_id)` 获取相关典籍摘要，注入 System Prompt。
2. **顿悟触发后**：`keeper.compile(insight_content, agent_id, ...)` 将新洞察编纂入库，并调用 `mark_source_agent` 标记归属。

其余所有复杂操作（全文加载、溯源、图谱扩散）均由 Agent 通过工具显式触发（例如内置的 `sutra_load` 工具，由核心引擎提供）。

## 二、核心模型

### 2.1 典籍 (Sutra)

知识的最小原子单元，一条典籍对应原始文档中的一个语义连贯片段。

```pseudocode
Sutra:
    id: UUID
    title: string                     // 自动或人工生成标题
    summary: string                   // 约 50–80 tokens 的摘要，供 L0 展示
    content: string                   // 全文（Level 1）
    source_location: SourceLocation   // 原文溯源信息
    tags: list<string>                // 关键词标签
    metadata:
        source_type: "document" | "epiphany" | "import"
        origin_doc_id: string         // 原始文档唯一标识（如果来自文档导入）
        created_by_agent: string      // 顿悟来源 Agent ID
        created_at: timestamp
        confidence: float             // 顿悟生成的置信度
    vector: float[]                    // 内容向量（用于语义检索）
```

### 2.2 原文溯源信息 (SourceLocation)

确保每一条典籍都能追溯到其知识来源，避免“知识孤岛”。

```pseudocode
SourceLocation:
    file_path: string?                // 本地文件绝对路径或相对路径
    url: string?                      // 网页原始链接
    start_line: number?               // 起始行号
    end_line: number?                 // 结束行号
    chapter: string?                  // 所属章节/段落标题
    snippet: string                   // 原始文本的前 200 字符，用于人工定位
```

当 Agent 使用 `sutra_load` 加载典籍全文时，自动附带 `source_location`；用户或 Agent 可通过 `locate_source` 工具直接跳转到原文位置。

### 2.3 经络链接 (Meridian Link)

典籍之间的语义关联边，构成知识图谱。

| 关系类型 | 含义 | 示例 |
|:---|:---|:---|
| `causes` | A 导致 B | “缓存未命中” → “响应延迟升高” |
| `solves` | A 解决 B | “索引优化” → “慢查询” |
| `evolves_from` | B 是 A 的演进版本 | “V2 架构” 演进自 “V1 架构” |
| `references` | A 引用 B 作为依据 | “SQL 调优” 引用 “执行计划分析” |
| `contradicts` | A 与 B 冲突 | “官方建议” 与 “社区实践” |

链接由系统在编纂时自动提取，也可人工标注。检索时可沿链路进行 1~2 跳扩散，召回间接相关知识。

### 2.4 典籍集 (Sutra Bundle)

一组领域相关的典籍的打包分发单元，类似于弈林中的“搭配包”。一个典籍集包含：

- 若干条典籍的全文 + 元数据 + 向量（可预计算）
- 经络链接的图结构导出
- `bundle.yaml` 清单文件

这使用户可以下载例如“PostgreSQL 调优知识包”、“React 最佳实践”等，直接注入藏经阁。

## 三、取经：检索引擎

### 3.1 检索流程总览

```
Agent 发起 query
    │
    ▼
┌─────────────────────┐
│ 混合检索 (Hybrid)    │
│  - 语义向量检索      │
│  - 关键词 BM25       │
│  - 标签精确过滤      │
└─────────┬───────────┘
          │ 初排 Top-K
          ▼
┌─────────────────────┐
│ RRF 融合排序         │
│ (Reciprocal Rank)   │
└─────────┬───────────┘
          │ 统一排名
          ▼
┌─────────────────────┐
│ 图谱扩散 (可选)       │
│ 沿经络链接扩展 1 跳   │
└─────────┬───────────┘
          │ 合并结果
          ▼
┌─────────────────────────────┐
│ 渐进式返回：                │
│  Level 0 → 摘要列表         │
│   (由外层循环注入 prompt)   │
└─────────────────────────────┘
          │
          ▼ (Agent 主动调用 sutra_load)
┌─────────────────────────────┐
│ Level 1 → 全文 + 溯源信息    │
└─────────────────────────────┘
```

### 3.2 混合检索实现

```pseudocode
FUNCTION hybrid_search(query, agent_id, top_k=5, depth=1):
    // 1. 语义检索
    query_vec = embed(query)
    semantic_ids = vector_db.search(
        collection="sutra_vectors",
        vector=query_vec,
        top_k=top_k * 2
    )
    
    // 2. 关键词检索 (BM25 或全文索引)
    keyword_ids = inverted_index.search(
        query=query,
        top_k=top_k * 2,
        filter_by_agent=agent_id  // 可选权限范围
    )
    
    // 3. 标签匹配（如果 query 中包含明确标签）
    tag_ids = tag_index.filter(query.tags)
    
    // 4. RRF 融合
    fused_scores = reciprocal_rank_fusion([
        (semantic_ids, weight=1.0),
        (keyword_ids, weight=0.8),
        (tag_ids, weight=0.5)
    ])
    top_sutra_ids = fused_scores.top(top_k)
    
    // 5. 图谱扩散（depth>0）
    IF depth > 0:
        expanded_ids = graph_db.expand(
            start_nodes=top_sutra_ids,
            hops=min(depth, 2),
            exclude_types=["contradicts"]  // 避免扩散冲突信息
        )
        top_sutra_ids = merge_and_rerank(top_sutra_ids, expanded_ids, top_k)
    
    // 6. 返回摘要
    RETURN [ get_summary(id) for id in top_sutra_ids ]
```

### 3.3 渐进式返回与溯源工具

核心引擎可提供两个工具给 Agent 调用，以完全融入技能系统：

```pseudocode
// Level 0 → Level 1 的桥梁
TOOL sutra_load:
    description: "加载一本典籍的全文和来源信息"
    parameters:
        - sutra_id: string
    
    EXECUTE:
        sutra = keeper.get_full_sutra(sutra_id, session.agent_id)
        session.context.append(
            f"[典籍] {sutra.title}\n"
            f"摘要: {sutra.summary}\n"
            f"原文: {sutra.full_content}\n"
            f"来源: {sutra.source_location.file_path}:{sutra.source_location.start_line}"
        )
        RETURN "已加载典籍"

// 溯源：直接定位原文位置
TOOL sutra_locate_source:
    description: "在原始文档中定位典籍引用的具体位置"
    parameters:
        - sutra_id: string
    
    EXECUTE:
        location = keeper.locate_source(sutra_id)
        // 在 IDE 中打开文件并跳转到行号
        editor.goto(location.file_path, location.start_line)
        RETURN f"已跳转到 {location.file_path}:{location.start_line}"
```

## 四、编纂：入库管道

### 4.1 文档导入编纂流程

当用户或 Agent 向藏经阁导入一篇文档时：

```
原始文档
    │
    ▼
┌──────────────┐
│ 语义切分       │ → 保留原文位置 (行号/段落)
│ (Semantic     │
│  Chunking)    │
└──────┬───────┘
       │ 每个 chunk
       ▼
┌──────────────┐
│ 提取标题/摘要  │ → LLM 调用 (轻量)
│ 自动打标签     │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 向量化         │ → embedding 模型
└──────┬───────┘
       │
       ▼
┌──────────────────┐
│ 入库并建立        │
│ - 向量索引        │
│ - 关键词倒排      │
│ - 经络链接(抽取)  │ → 基于实体关系抽取或启发式规则
└──────────────────┘
```

### 4.2 语义切分策略

使用**语义边界切分**而非固定长度：

- 优先在标题、列表、段落边界断开。
- 保持每个典籍的 token 长度在 300~800 之间，适合一次上下文注入。
- 记录 `start_line`、`end_line` 保证溯源精度。

### 4.3 经络链接自动抽取

利用轻量级 NER + 规则模板，或调用一次小参数 LLM 判断两个典籍间的关系。仅在出现强信号（如显式引用、标题相似）时才建立链接，避免图谱过密。

## 五、顿悟闭环

### 5.1 与进化引擎的配合

进化引擎在顿悟触发后，调用 `keeper.compile()` 将新洞察入库。细节如下：

```pseudocode
// 进化引擎中的逻辑（已在 v3.1 中定义，这里补充细节）
FUNCTION trigger_epiphany(trajectory, agent_id, keeper):
    IF keeper IS NoOpSutraKeeper:
        // 只生成 Skill，不入库
        generate_skill_only(...)
        RETURN
    
    insight = extract_insight(trajectory)
    
    // 1. 去重检测
    similar = keeper.query(
        query = insight.title,
        agent_id = agent_id,
        depth = 0,
        top_k = 3
    )
    IF similar.has_high_similarity(threshold=0.92):
        // 更新现有典籍的链接和引用权重
        keeper.strengthen_meridian(insight, similar.best_match)
        RETURN "经络更新"
    
    // 2. 编纂为新典籍
    new_ids = keeper.compile(
        document = format_as_document(insight),
        agent_id = agent_id,
        metadata = {
            source_type: "epiphany",
            trajectory_id: trajectory.id,
            confidence: calculate_confidence(trajectory)
        }
    )
    
    // 3. 标记归属（该典籍由此 Agent 创造）
    keeper.mark_source_agent(new_ids, agent_id)
    
    // 4. 同时生成 Skill（原有逻辑）
    generate_skill_from_insight(insight, agent_id)
```

### 5.2 置信度与去重

顿悟生成的典籍带有 `confidence` 属性，可在检索时降低权重或标注为“来自反思”。去重使用语义相似度（向量余弦相似度 > 0.92 且标题/关键词高度重合）判断是否已存在类似知识。

## 六、存储架构

### 6.1 推荐技术栈

| 组件 | 推荐实现 | 说明 |
|:---|:---|:---|
| 向量数据库 | Chroma / Qdrant / Milvus Lite | 存储和检索典籍向量 |
| 全文索引 | SQLite FTS5 / Tantivy | 轻量关键词倒排 |
| 图数据库 | NetworkX (本地) / Neo4j (可选) | 存储经络链接图 |
| 原文存储 | 本地文件系统 + 元数据 JSON | 原文文件保留在 `sutras/originals/` |
| 嵌入模型 | text-embedding-3-small / bge-large | 生成向量 |

所有组件可内嵌运行，无需外部服务，保证离线可用。

### 6.2 数据生命周期

- 典籍**不可变性**：一旦入库，内容只读。更新视为新版本，旧版保留并建立 `evolves_from` 链接。
- 索引定期重建优化（低优先级）。
- 用户可手动剔除低质量典籍。

## 七、接口契约

完整接口已在主纲领附录 A 中定义，此处仅做关键复述并补充细节：

```pseudocode
interface ISutraKeeper:
    // 取经：查询 (L0 摘要)
    FUNCTION query(query: string, agent_id: string, depth: number DEFAULT 1, top_k: number DEFAULT 5) -> SutraQueryResult
    
    // 取经：获取全文 (L1)
    FUNCTION get_full_sutra(sutra_id: string, agent_id: string) -> SutraFullContent
    
    // 取经：溯源定位
    FUNCTION locate_source(sutra_id: string, fragment: string OPTIONAL) -> SourceLocation
    
    // 编纂：将文档/洞察入库
    FUNCTION compile(document: Document, agent_id: string, metadata: Metadata) -> list<string>
    
    // 管理：标记典籍归属于某个 Agent
    FUNCTION mark_source_agent(sutra_ids: list<string>, agent_id: string) -> void
    
    // (新增) 管理：加强经络链接
    FUNCTION strengthen_meridian(insight: EpiphanyInsight, existing_sutra_id: string) -> void
    
    // (新增) 导出：导出一个典籍集包
    FUNCTION export_bundle(bundle_name: string, sutra_ids: list<string>) -> BundleManifest
```

## 八、与弈林市场集成

### 8.1 典籍集组件

在弈林市场中，知识库可以作为一种新的组件类型：`knowledge` 或 `sutra-bundle`。清单示例：

```yaml
# package.yaml
name: knowledge/postgres-tuning-guide
version: 1.0.0
type: sutra-bundle
author: "DB Team"
description: "PostgreSQL 性能调优知识典籍集，含 120 条典籍及经络链接"
dependencies: {}   # 典籍集一般不依赖其他组件
included_files:
  - sutras.json          # 所有典籍的全文及元数据导出
  - vectors.bin          # 预计算的向量（可选，允许重新计算）
  - meridian_graph.json  # 经络链接导出
```

### 8.2 安装流程

当用户执行 `yilin install knowledge/postgres-tuning-guide`：

1. 下载包。
2. 调用 `keeper.import_bundle()` 方法将典籍注入本地藏经阁。
3. 若该 Agent 未启用藏经阁（`keeper` 为 `NoOp`），则提示用户需先初始化藏经阁存储。

安装后，所有绑定了藏经阁的 Agent 均可检索到这些典籍，除非设置访问权限（未来可扩展）。
