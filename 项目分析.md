### 3. 系统提示词 (System Prompt) 构建

系统提示词在 **`outer.ts`** (第 97-135 行) 中通过 `assembleStaticPrompt()` 函数组装，流程如下：

#### 3a. 组装流程 (`outer.ts:97-135`)

```typescript
function assembleStaticPrompt(charMeta, charContent, toolDefs, workspace): string {
  // 1. 角色灵魂 (Soul) — 来自 data/characters/{id}/soul.md
  if (charContent.soul) parts.push(`## Character\n${charContent.soul}`)
  
  // 2. 用户信息 (User Profile) — 来自 data/characters/{id}/user.md  
  if (charContent.user) parts.push(`## User Info\n${charContent.user}`)
  
  // 3. 可配置提示模板 — 来自 data/prompts/default.md (或角色专属 prompt.md)
  parts.push(loadPromptTemplate(charMeta.id).trim())
  
  // 4. 可用工具列表 — 排序后列出名称+描述
  if (toolDefs.length > 0) {
    const sorted = normalizeTools(toolDefs)
    parts.push(`## Available Tools\n${toolListings.join('\n')}`)
  }
  
  // 5. 技能索引 (只有名称+描述, 不含正文) — 来自 data/skills/
  const skillIndex = buildSkillIndex(charMeta)
  // ... 添加 "## Available Skills" 和 "## Skill Attachments"
  
  // 6. 工作区路径
  parts.push(`## Workspace\nYou can access: ${workspace}`)
}
```

#### 3b. 提示模板 (`data/prompts/default.md`)

约 33 行的默认模板包含：
- **System Prompt** (平台标识 + 管理规则)
- **Tool Use** 指南
- **Knowledge and Search** 指南
- **Copyright** 政策
- **Safety** 指导
- **Execution Rules** (继续执行、批量读取、先行动、真相验证)

#### 3c. 提示词注入 (messages 数组构建, `outer.ts:488-516`)

最终的消息数组包含多个系统消息以实现前缀缓存优化：

```typescript
// 消息 [0]: 静态系统提示词 (可被缓存)
[{ role: 'system', content: systemPrompt }]
// 消息 [1] (可选): 记忆 — 独立系统消息, 不失效缓存
if (charContent.memory) { role: 'system', content: `## Memory\n${charContent.memory}` }
// 消息 [2] (可选): 压缩历史摘要 — 单独的 [Compacted History] 块
if (summary) { role: 'system', content: `[Compacted History]\n${summary}` }
// 后续: 对话历史 + 时间戳注入
// 每轮前注入当前时间: { role: 'system', content: `[Current time: ...]` }
```

**关键设计决策** (注释在 `outer.ts:487`)：
> Memory is always a separate system message (msg[1]) so it never invalidates the prefix cache

### 4. LLM 消息流

完整的消息流（用户输入 → LLM 响应 → 工具执行）：

#### 步骤 1: 客户端发起 (`web/client/src/stores/chat.ts:453-491`)
- `sendMessage()` 构建消息，初始化 Socket.IO 监听器
- 通过 `socket.emit('chat-run', {...})` 发送到服务端
- 附带: session_id, character_id, input, model, provider_id, workspace, thinking 参数等

#### 步骤 2: 服务端接收 (`ws/chat.ts:21-70`)
- `registerChatSocket` 处理 `chat-run` 事件
- 创建/更新会话，保存用户消息到数据库
- 调用 `enqueueRun(sessionId, callback)` 将运行加入队列

#### 步骤 3: 外循环启动 (`agent/session-runner.ts`)
- 会话排程器管理运行状态 (idle/running)
- `sessionLoop()` 在 `outer.ts:368` 开始

#### 步骤 4: 外循环 (outer.ts:368-846)
1. **加载配置**: 会话、角色、提供者、模型、工具定义
2. **MCP 连接**: 连接到角色绑定的 MCP 服务器（带重试）
3. **构建系统提示词**: 组装静态提示词（带缓存）
4. **构建消息数组**: 系统 + 记忆 + 压缩历史 + 数据库历史（每轮注入时间戳）
5. **上下文压缩检查**: 如果接近限制，进行 LLM 驱动的摘要
6. **进入内循环**: 每轮调用 `innerLoop()`

#### 步骤 5: 内循环 (inner.ts:193-470)
1. **LLM 调用** (`streamWithRetry`, 第 114-191 行):
   - 调用 `streamChatCompletion()` 最多 3 次（含指数退避重试）
2. **流式响应处理**:
   - 文本增量 → 通过 `message.delta` 事件发送到前端
   - 推理内容 → 通过 `message.delta` (reasoning) 发送到前端
   - 工具调用 → 累积为 `toolCallsAcc[]`
   - 用量信息 → 通过 `usage` 事件发送到前端
3. **信号检测**:
   - `delegate_task` → 返回 `SubAgentRequestData` 给外循环
   - `task_complete` → 结束会话，附带摘要
4. **工具执行** (两阶段)：
   - **L1**: `checkToolBinding()` — 工具在角色白名单中且约束满足吗？
   - **L2**: `checkStrategy()` — Plan 模式拒绝危险工具；Ask 模式需用户确认
   - **执行**: 只读工具并行，写入工具串行
   - 结果 → 格式化为 tool 角色消息 → 追加到数组

#### 步骤 6: 外循环后处理
- Sub-agent 处理：`spawnAndRunSubAgent()` 拉起独立会话 → 回收摘要
- 上下文压缩：如果超过 75% 限制，LLM 总结并替换早期消息
- 执行演化检测
- 清理 MCP 连接

### 5. LLM 提供者 API 调用 (`llm/client.ts`)

所有提供者通过单一通用客户端通信：

**文件**: `C:\Users\dmql\Documents\Yi\Yi-Lin\web\server\src\llm\client.ts`

#### `streamChatCompletion()` (第 52-171 行)

```typescript
async function* streamChatCompletion(opts: LLMOptions): AsyncGenerator<LLMChunk>
```

- **端点**: `${baseUrl}/chat/completions`（符合 OpenAI API 格式）
- **方法**: `POST` 携带 `Authorization: Bearer {apiKey}`
- **请求体**: `{ model, messages, stream: true, stream_options: { include_usage: true }, tools?, thinking?, reasoning_effort? }`
- **响应处理**: 手动 SSE 解析（非 SDK），缓冲行并按 `data: ` 前缀解析
- **产出的事件**: `delta` (文本/推理/工具调用增量), `done` (含 finish_reason + usage), `error`, `usage` (流式中间结果)
- **缓存字段**: 从提供者提取 `prompt_cache_hit_tokens`, `prompt_cache_miss_tokens`, `prompt_tokens_details.cached_tokens`

#### 提供者配置 (`data/providers.json`)

三个提供者:
1. **OpenCode Go** (`https://opencode.ai/zen/go/v1/`) — 20 个模型（MiniMax, Kimi, GLM, DeepSeek, Qwen, Mimo 等）
2. **OpenCode Zen** (`https://opencode.ai/zen/v1/`) — 44 个模型（Claude, Gemini, GPT, Grok, DeepSeek, GLM, MiniMax, Kimi, Qwen 等）
3. **Deepseek** (`https://api.deepseek.com`) — 2 个模型（deepseek-v4-flash, deepseek-v4-pro）

**所有提供者共享同一端点格式** — 当前无 OpenAI、Anthropic 或任何其他 SDK 的专用代码路径。

### 6. 缓存系统

系统提示词缓存分为两层 — 内存 + 磁盘：

#### 6a. 系统提示词缓存 (`agent/system-cache.ts`)

**文件**: `C:\Users\dmql\Documents\Yi\Yi-Lin\web\server\src\agent\system-cache.ts`

**核心机制**:
- **键生成** (`stableKey()`, 第 28-46 行): 来自 `{ characterId, tools(排序后), skills(排序后), soul(hash), user(hash) }` 的 SHA-256 指纹
- **内存缓存**: 最多 50 个条目，LRU 驱逐
- **磁盘缓存**: 文件存储在 `.cache/system-prompt/{key}.json`
- **写时去重**: 检查内存中的哈希，跳过相同内容的写入 (第 78 行)
- **诊断**: `diagnoseMiss()` 跟踪哪些组件变更（工具/技能/灵魂/用户）触发了缓存未命中

**在 outer.ts 中使用** (第 471-485 行):
```typescript
const key = stableKey(charMeta.id, normalizeTools(toolDefs), charMeta.skills, charContent.soul, charContent.user)
let systemPrompt = getCached(key)
if (!systemPrompt) {
  // 日志未命中原因，组装，设置缓存
  systemPrompt = assembleStaticPrompt(charMeta, charContent, toolDefs, ...)
  setCached(key, systemPrompt)
}
```

#### 6b. LLM 端缓存核算 (`llm/client.ts`, 第 111-121 行)

LLM 客户端从提供者响应中提取缓存字段：
```typescript
// 提供商缓存命中/未命中 token
usage.cache_hit_tokens = u.prompt_cache_hit_tokens  // Anthropic 风格
usage.cache_miss_tokens = u.prompt_cache_miss_tokens
// 或 OpenAI 风格:
usage.cache_hit_tokens = u.prompt_tokens_details?.cached_tokens
```

整个运行过程在 `outer.ts:607-608` 累计：
```typescript
totalCacheHitTokens += result.totalCacheHitTokens
totalCacheMissTokens += result.totalCacheMissTokens
```

运行结束时记录命中率（`outer.ts:783-786`）:
```
[cache] {sessionId}: hit={totalCacheHitTokens} miss={totalCacheMissTokens} ratio={hitRatio}%
```

#### 6c. 提示词前缀缓存的架构设计

系统提示词中**记忆是单独的系统消息**（`outer.ts:489-491`），这样静态提示词（消息[0]）保持不变，提供商的提示前缀缓存跨轮保持有效。这种设计明确记录在 `outer.ts:487` 的注释中。

### 12. 关键设计模式

1. **缓存优先的系统提示词**: 提示词组装 + 指纹 + 内存/磁盘缓存，记忆作为独立系统消息与前缀缓存解耦
2. **双循环 Agent 引擎**: 外循环（会话生命周期）→ 内循环（每个 LLM 轮次）→ sub-agent 分支返回外循环
3. **两阶段权限**: L1（绑定 + 约束校验）+ L2（策略拦截：Plan/Ask/Bypass）
4. **工具并行性**: 只读工具（read, grep, glob, webfetch）并行执行；写入工具串行
5. **SSE 原生，无 SDK**: 整个 LLM 通信层是带有手动 SSE 解析的 ~170 行 TS 文件，跨所有提供者兼容 OpenAI API 格式
6. **上下文压缩**: LLM 驱动的摘要（限 8K tokens 保留率），冷恢复（>24h）和溢出恢复
7. **事件驱动演化**: 洞察检测 → 创建事件 → 调度执行 → 技能生成
