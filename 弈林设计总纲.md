## 一、弈林概述

### 1.1 定位

弈林是“弈”系统的**生态市场**，并已演化为**群体智能的协调层**。它解决的核心问题从“复用与分享”拓展为 **AI 能力的供需匹配与集体进化**：

- **发现**：开发者不再需要到各处论坛、GitHub、Discord 搜寻 Skill/MCP/插件，弈林提供中心化排行榜、分类、评分，所有能力一目了然。
- **装配**：用户可直接预览某个 Agent 角色、技能、工具组合，甚至是一整套知识库，**一键安装到本地引擎**，组件间依赖自动解析，绑定自动完成。
- **进化**：每个用户本地的 Skill 根据实战顿悟自动优化。通过**合流 (Confluence)** 机制，不同用户的改进版本可上传为提案，经人工审核合并后形成官方新版本，惠及全体使用者。
- **交易（规划中）**：高质量知识库、专家技能将被允许设定价格，创作者获得收益。知识本身成为可流通资产，推动专业化分工。

弈林**不修改核心引擎**，所有组件最终体现为 `AgentRegistration`、`Skill`、`ToolBinding`、`Sutra` 等标准结构，只是来源变为市场，且市场成为进化的枢纽。

### 1.2 协同进化闭环

```
创作者 ──发布──→ 弈林市场 ──安装──→ 用户A、B、C...
  ↑                    │
  │                    ├─ 用户各自使用，本地顿悟生成变体
  │                    │   └─ 上传提案 (Proposal)
  │                    ├─ 市场聚合展示提案
  │                    ├─ 维护者人工合并 → 新版本发布
  │                    └─ 用户更新，获得进化后的能力
  └────────── 下载量/评分/收益 ────────────────┘
```

**百川合流，生生不息**：Skill 不再是静态的文档，而是一个持续演化的生命体。五个用户、五次实战、五次顿悟，最终汇聚成一次集体智慧的版本跃迁。

## 二、核心概念

### 2.1 组件类型

| 类型 | 标识符 | 说明 | 对应主系统实体 |
|:---|:---|:---|:---|
| **role** | 角色 | 包含 SOUL.md、MEMORY.md 等完整身份定义，可声明依赖的技能/工具/MCP | `roles/<name>/` 目录 + `agents.yaml` 条目 |
| **group** | 组 | 一组 Agent 的协作拓扑（组名、成员角色引用） | `registry/agents.yaml` 中的多条 agent 定义 + 组约束 |
| **skill** | 技能 | 单个 SKILL.md 及附属文件，**支持合流进化** | `skills/<name>/` 目录 |
| **tool** | 工具绑定 | 工具定义 + 约束 (ToolBinding) | `registry/agents.yaml` 中的 `tools` 段 |
| **mcp** | MCP 连接 | MCP 服务器连接配置 | 全局 MCP 配置引用 |
| **knowledge** | 知识库 | 一组已索引的典籍集合（文档 + 向量 + 经络图谱） | 藏经阁 (SutraKeeper) 中的典籍库 |

**合流进化目前主要应用于 Skill 组件**，未来可扩展至角色和知识库。

### 2.2 排行榜

弈林提供多维度的排行榜，帮助用户快速发现高质量组件：

- **热门总榜**：综合下载量、安装量、评分、最近活跃度排序。
- **分类榜**：按角色、技能、知识库等类型分别排行。
- **趋势榜**：最近一周安装增长最快的组件。
- **新秀榜**：新发布但评价极高的组件。
- **进化活跃榜**：合流提案数量最多、合并最频繁的组件（反映社区活跃度）。

### 2.3 合流提案 (Confluence Proposal)

一个**提案**代表某用户本地生成的 Skill 改进版本。它包含：

- 基于上游版本 (base_version)
- 变更摘要（AI 生成的自然语言 diff 描述）
- 改进统计（成功次数、自我纠正次数等）
- 优化后的完整 Skill 内容
- 提案人及信誉信息

提案上传后，在弈林中该 Skill 的详情页可查看所有待合并提案，**由维护者（作者或指定管理者）审核合并**。

### 2.4 知识交易预留

在包模型和接口层预留交易扩展字段，但**v1.3 不实现支付与结算**。

## 三、包模型

### 3.1 知识库组件包结构

```
react-best-practices/           # knowledge 组件
├── package.yaml
├── documents/                  # 原始文档（Markdown、PDF 等）
│   ├── hooks.md
│   ├── state-management.md
│   └── performance.md
├── prebuilt/                   # 可选：预构建的向量索引与元数据
│   ├── embeddings.bin
│   ├── sutra_metadata.json
│   └── graph_edges.json
└── README.md                   # 知识库说明
```

`package.yaml` 示例：

```yaml
name: knowledge/react-best-practices
version: 2.1.0
type: knowledge
author: "ReactExpert"
description: "React 18 常见模式、性能优化与状态管理最佳实践"
price: free                       # 或 "5.99 USD" (暂不生效)
license: CC BY-SA 4.0
tags: [react, frontend, best-practices]
knowledge_domain: "react"
included_sutras_count: 24
has_prebuilt_index: true
```

### 3.2 其他组件的交易字段

非知识库组件同样可声明 `price`，为未来全面交易铺路。

### 3.3 Skill 组件包结构扩展

Skill 组件包内增加 `.evolved/` 目录用于存储本地变体（不上传到公共注册表）：

```
skill-name/
├── package.yaml
├── SKILL.md                   # 原始基线内容（安装时的版本）
├── assets/
└── .evolved/                  # 本地变体区（不上传到市场）
    ├── variants.json          # 记录所有本地变体元数据
    └── variant_001.md         # 顿悟生成的优化版本
```

`variants.json` 示例：

```json
{
  "variants": [
    {
      "id": "v_001",
      "base_version": "1.2.0",
      "created_at": "2026-07-03",
      "description": "增加对大文件处理的并行优化步骤，改善错误恢复逻辑",
      "success_count": 4,
      "correction_count": 1,
      "file": "variant_001.md"
    }
  ]
}
```

## 四、市场架构

### 4.1 排行榜 API

远程市场提供排行榜查询端点：

```http
GET /rankings?type=role&period=week&limit=20
# 返回排名列表，包含组件摘要、评分、下载量等
```

CLI 示例：

```bash
yilin top skill              # 技能榜
yilin top knowledge          # 知识库榜
yilin trending               # 趋势榜
yilin show role/lead-architect   # 展示详情，含评价、安装量
```

### 4.2 合流提案相关 API

```http
GET /component/skill/xxx/proposals   # 列出某 Skill 的所有合流提案
POST /component/skill/xxx/proposals  # 上传提案
```

CLI 命令：

```bash
yilin evolve push skill/xxx      # 上传本地变体为提案
yilin evolve status skill/xxx    # 查看该 Skill 的提案列表
yilin top evolved                # 查看进化活跃榜
```

### 4.3 一键集成流程

用户浏览某个 Agent 角色（如“全栈架构师”）：

```
$ yilin show role/fullstack-lead

名称: role/fullstack-lead
版本: 3.2.1
评分: ★★★★☆ (4.8/5, 230 评价)
安装量: 12.3k
简介: 包含 React+Node.js 的架构师角色，内置 CRUD 生成技能。
自动绑定组件:
  - skill/crud-generator
  - tool/postgres-schema-reader
  - mcp/playground-db

命令: yilin install role/fullstack-lead
```

安装一个命令，角色、绑定技能、工具、MCP 全部配置完毕，Agent 立即可用。

### 4.4 发现能力闭环

- 安装角色时，系统自动推荐“使用此角色的用户也安装了哪些技能/知识库”。
- 知识库可被标记为“适配某角色”，如 `knowledge/react-best-practices` 适配 `role/frontend-dev`，安装角色时可一键关联。

## 五、安装与集成

### 5.1 知识库安装流程

执行 `yilin install knowledge/react-best-practices` 后：

1. 下载组件包。
2. **如果没有预构建索引**：调用藏经阁 `ISutraKeeper.compile()` 接口，将 `documents/` 下的原始文档逐一**语义切分、生成典籍、建立索引**。这可能需要一定时间和计算资源，安装时显示进度。
3. **如果有预构建索引**：直接导入 `prebuilt/` 中的向量、元数据和图谱，跳过重索引，实现秒级安装。
4. 典籍入库后，通过 `mark_source_agent` 将这批典籍与用户的某个或某些 Agent 关联（默认关联给当前活跃 Agent，或由用户选择）。
5. 更新 `.yilin/lock.yaml`，记录知识库包版本与来源。

**重要**：知识库的安装不修改核心引擎代码，完全通过 `ISutraKeeper` 接口操作藏经阁。如果用户尚未启用藏经阁模块，安装知识库时弈林会提示需要先启用藏经阁。

### 5.2 角色依赖自动安装

角色安装时，弈林解析其 `suggested_bindings` 与 `dependencies` 字段，自动拉取并配置：

- 所需技能放入 `skills/` 并绑定到该 agent。
- 工具绑定配置写入 `agents.yaml` 的 `bindings.tools` 段。
- MCP 连接注册到全局配置并在 agent 内引用。
- 若依赖其他角色或知识库，同样递归安装。

用户可在安装后随时调整绑定，弈林不会强制覆盖已有配置。

### 5.3 合流更新

当用户执行 `yilin update skill/xxx` 时，弈林会检查上游是否有新版本（可能包含合流合并的结果）。若有，下载新版本并提示变更内容。用户可以选择：

- **直接更新**：用新版覆盖本地 Skill。
- **保留本地变体**：将当前本地变体自动备份至 `.evolved/`，然后更新。
- **先合并再更新**：若本地也有未上传的变体，可先上传提案，等上游合并后再更新。

更新后，核心引擎的 `skill_load` 机制无需改变，因为 Skill 文件路径不变，只是内容更新。Agent 下次加载时自然获得最新能力。

## 六、发布流程

### 6.1 发布知识库

创作者准备知识库包，可选两种模式：

- **源文档模式**：只包含原始文档，由用户端索引（适合文档变动频繁的场景）。
- **预索引模式**：本地运行索引工具生成 `prebuilt/` 目录，打包发布。用户安装时无需额外计算，体验更好。

发布命令：

```bash
yilin pack ./my-knowledge           # 打包
yilin publish ./my-knowledge        # 上传
```

上传后，市场服务自动提取典籍数量、领域标签等信息用于搜索和排行。

### 6.2 发布其他组件

角色、技能、工具绑定、MCP 组件遵循类似流程，确保 `package.yaml` 完整，打包后上传。合流提案上传通过专门的 `evolve push` 命令进行。

## 七、安全与信任

### 7.1 知识库内容审查

知识库包中可能包含任意文本，甚至恶意指令。弈林采取：

- 安装前预览功能：`yilin inspect knowledge/xxx` 可查看包内文档摘要，不直接索引。
- 可选的社区标记系统：标记包含推广、低质或违规内容的知识库。
- 未来交易场景：引入创作者实名/认证，付费知识库需经过内容审核。

### 7.2 交易安全预留

交易接口设计时需考虑：

- 授权验证：安装付费知识库时，需验证购买凭证。
- 防二次分发：典籍可嵌入轻量水印（版权声明），但不做硬性 DRM（与开放理念平衡）。
- 退费与争议：接口层预留，但不纳入 v1.3 实现。

### 7.3 合流提案审查

为避免恶意提案（如植入有害指令），弈林采取多层防护：

- **自动扫描**：对提案中的 Skill 内容进行关键词和模式检测，发现潜在的 `rm -rf` 等高危指令自动标记。
- **社区信誉**：提案人历史提案的采纳率构成信誉分。低信誉用户的提案需更多人工或社区审核。
- **维护者人工审核**：默认所有提案由原始作者（或指定维护者）人工审核，可参考社区投票。**当前版本不启用自动合并**，确保每一步质量可控。
- **用户本地终审**：即使提案已合并入官方版本，用户更新时仍可预览新旧对比，自行决定是否采纳。

## 八、合流进化 (Confluence)

> *合流者，众水所汇也。五方之智，各擅胜场；纳而融之，乃成江海。*

### 8.1 设计原则

- **本地自治**：顿悟进化依然在用户本地完成，由“弈”核心进化引擎触发，不依赖网络。
- **显式分享**：本地优化不会自动上传，必须由用户显式执行 `yilin evolve push`。
- **溯源清晰**：每个提案记录基版本、生成环境、成功轨迹，确保可审计。
- **人工把关**：当前合并由维护者人工审核确认，未来可能引入辅助自动合并机制。
- **冲突可解**：多人修改同一部分时，合并流程利用 LLM 辅助调和，必要时由维护者裁定。

### 8.2 本地变体生成（修改进化引擎行为）

当进化引擎的顿悟条件满足且决定生成新 Skill 时，行为从“直接覆盖本地文件”改为：

1. 读取当前 Skill 的基版本信息（从 `package.yaml` 或内置元数据获取）。
2. 生成优化后的 Skill 完整内容。
3. 写入 `.evolved/variant_{id}.md`。
4. 在 `variants.json` 中登记变体元数据。
5. **原 Skill 文件保持不变**。Agent 在加载 Skill 时，可通过配置项选择使用“原版”还是“最新本地变体”（默认使用最新变体）。

这样既保留了原始安装版本作为回溯基准，又允许多个变体并存。

### 8.3 提案上传

```bash
yilin evolve push skill/xxx --variant latest
```

- 弈林读取指定变体的内容、元数据、基版本。
- 请求用户输入简短的变更说明（可选，默认用 AI 生成的摘要）。
- 打包提案（包含完整 Skill 内容、元数据）上传至市场对应 Skill 的提案区。
- 上传成功后，该变体被标记为“已提交”，避免重复上传。

### 8.4 市场端提案管理与合并

1. **提案展示**：在 Skill 详情页以时间线或列表形式展示所有提案，包含 diff 对比视图（由市场服务端计算或由前端渲染）。
2. **讨论与投票**：其他用户可对提案进行评论和投票（+1/-1），作为维护者决策的参考。
3. **合并决策（当前：人工合并）**  
   当前版本**仅支持人工合并**。具体流程：
   - 维护者（通常为 Skill 原始作者或指定的社区维护者）在后台查看提案。
   - 系统辅助展示：提案变更摘要、社区投票结果、基于基版本的自动冲突分析。
   - 维护者判断后，手动选择要合并的提案，触发合并。
   - 合并过程使用合并引擎自动合成，但最终**由维护者审阅并确认新版本内容**，方可发布。
4. **合并引擎**：
   - 以原始 Skill 为基础，应用选定提案的修改。
   - 若多个提案修改同一部分，利用 LLM 分析语义冲突并尝试调和，生成合并建议（如标注冲突区域、提供合并选项）。
   - 维护者确认后，生成新版本 Skill（如 v1.3.0），存入市场作为正式版本。
   - 自动生成变更日志，列出本次合流整合了哪几个提案。
5. **通知与更新**：新版本发布后，所有安装该 Skill 的用户在 `yilin update` 时可见。
6. **未来方向：自动合并的可能性**  
   在社区信用体系成熟、提案质量普遍较高后，可引入**可选的自动合并机制**，例如：
   - 当提案满足「无冲突 + 社区投票超过阈值 + 提案人信誉分高于设定值」时，系统可自动触发合并。
   - 用户本地也可配置“自动采纳信誉分极高提案”的策略，减少手动更新步骤。
   该机制将在未来版本中根据社区反馈逐步开放，始终保留用户的手动干预权限。

### 8.5 版本谱系

每次合流合并，都会在 Skill 的版本历史中记录“合流节点”，形成图谱。这为后续的进化分析（哪些社区贡献最有效）提供数据基础。

## 九、目录结构变化

主项目根目录下，`skills/` 目录增加 `.evolved/` 子目录：

```
skills/
└── my-skill/
    ├── package.yaml
    ├── SKILL.md
    ├── assets/
    └── .evolved/              # 新增：本地变体
        ├── variants.json
        └── variant_001.md
```

核心引擎加载 Skill 时，通过 Skill 加载器读取变体配置，按优先级选择使用最新变体或基线版本。

此外，`.yilin/` 目录结构保持不变，但新增对合流提案状态、更新日志的缓存记录。

### 10.2 与藏经阁的交互

安装知识库时，弈林调用已经实现的 `ISutraKeeper` 接口（如果用户启用了藏经阁模块）。`compile()` 方法可直接支持批量文档入库。

### 10.3 与进化引擎的交互

进化引擎在触发顿悟时，调用一个本地回调（由弈林注入）将优化后的 Skill 内容保存为变体，而不再直接覆写原文件。此交互通过简单的文件写入和 JSON 记录实现，不引入运行时依赖。
